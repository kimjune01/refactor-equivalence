## Finding F1 — Moving py_repositories before protobuf creates the wrong protobuf repo first
**Severity**: blocker
**Claim**: C1
**What**: The claim says to move the entire `rules_python` block, including `py_repositories()`, before the explicit `com_google_protobuf` archive. In rules_python 1.9.0, `py_repositories()` conditionally creates a repository named `com_google_protobuf` if it does not already exist. If C1 is applied literally, `py_repositories()` runs before the WORKSPACE's explicit protobuf v31.0 archive, so it can predefine `com_google_protobuf` as rules_python's protobuf 29.0-rc2 dependency. The later explicit `http_archive(name = "com_google_protobuf", ...)` would then either fail as a duplicate repository definition or no longer be the repository consumed by later loads, breaking the protobuf v31.0 upgrade goal.
**Evidence**: `WORKSPACE:41` currently defines `com_google_protobuf` before the existing `rules_python` block at `WORKSPACE:111`. The upstream rules_python 1.9.0 `python/private/py_repositories.bzl` implementation contains `http_archive(name = "com_google_protobuf", ... url = "https://github.com/protocolbuffers/protobuf/releases/download/v29.0-rc2/protobuf-29.0-rc2.zip", ...)` guarded only by `maybe`, so it only stays inert if `com_google_protobuf` is already defined.
**Fix**: Remove C1, or narrow it so `py_repositories()` remains after the explicit `com_google_protobuf` archive has been declared.
