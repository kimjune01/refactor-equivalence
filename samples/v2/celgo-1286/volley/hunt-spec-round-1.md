## Finding F1 — C1 centralizes the wrong proto field-name rule
**Severity**: blocker
**Claim**: C1
**What**: The claim tells the implementer to centralize a `field.TextName()` / `field.JSONName()` conditional and use it from both `fieldToVariable` and `ContextProtoVars`. That changes `fieldToVariable`'s non-JSON behavior from `string(field.Name())` to `field.TextName()`. In this codebase, proto field lookup is indexed by descriptor `Name()`, not text-format name. The two differ for proto2 group fields, so the claim would create declarations keyed by `NestedGroup` while the registry/type model and existing tests expect `nestedgroup`. It also leaves `ContextProtoVars` looking up the registry field by `TextName()`, which cannot find a group field indexed by `Name()`.
**Evidence**: `cel/options.go:783` currently declares context variables with `string(field.Name())` unless JSON names are enabled, while `cel/options.go:869` uses `field.TextName()` for activation names. `common/types/pb/type.go:52` indexes field descriptions by `fd.Name()`, and `common/types/pb/type.go:346` defines that as `string(fd.desc.Name())`. `common/types/pb/type_test.go:97` asserts that the proto2 group field is found as `nestedgroup`. The protobuf API documents `TextName()` as different for groups at `vendor/google.golang.org/protobuf/reflect/protoreflect/type.go:305`.
**Fix**: Narrow C1 so the helper returns `string(field.Name())` in proto-name mode and `field.JSONName()` in JSON-name mode, and use that same helper in `ContextProtoVars` before `FindStructFieldType` and when populating `vars`.
