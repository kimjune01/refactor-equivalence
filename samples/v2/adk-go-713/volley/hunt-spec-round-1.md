## Finding F1 — Request parse helper does not specify error classification
**Severity**: warning
**Claim**: C2
**What**: The proposed helper shape `parseEventarcRequest(r *http.Request) (models.EventarcTriggerRequest, error)` does not say how the caller should distinguish structured-mode JSON decode failures from binary-mode body read failures. Current behavior uses two different `http.Error` responses: bad structured JSON returns status 400 with body text `Bad Request`, while body read failure returns status 500 with body text `Failed to read body`. With only a generic `error` return, an implementer has to guess whether to use sentinel errors, a typed error carrying status/message, additional return values, or to handle the HTTP response inside the helper.
**Evidence**: `round-1-claims.md:11` asks to preserve bad-JSON and body-read error handling at the call site while suggesting only `(models.EventarcTriggerRequest, error)`. Current behavior is in `server/adkrest/controllers/triggers/eventarc.go:64-66` and `server/adkrest/controllers/triggers/eventarc.go:79-82`.
**Fix**: Clarify the helper contract: either return enough structured information to preserve the exact status/body choices at the call site, or require the helper to use sentinel/typed errors that the handler maps back to the existing `http.Error(w, "Bad Request", http.StatusBadRequest)` and `http.Error(w, "Failed to read body", http.StatusInternalServerError)` responses.
