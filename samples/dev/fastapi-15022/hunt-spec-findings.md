- `volley-round1.md:21` and `volley-round1.md:23` conflict for explicitly streamed, annotated endpoints. Claim 9 requires `response_class=StreamingResponse` generator endpoints to work by passing the generator through as `content`, but claim 10 says `response_model` is set to `None` only when the route response class is the default placeholder and otherwise existing response-model inference is unchanged. For an endpoint like `async def stream() -> AsyncIterable[str]` with `response_class=StreamingResponse`, following claim 10 literally leaves FastAPI trying to infer/create a response model for `AsyncIterable[str]`, which is exactly the docs/test shape in `docs_src/stream_data/tutorial001_py310.py`. The spec needs to require suppressing response-model inference for explicit raw streaming annotations too, or explicitly reject that supported test shape.

- `volley-round1.md:27` makes `APIRoute.is_json_stream` depend only on the callable being a generator/async-generator and the response class being default, while `volley-round1.md:23` only creates `stream_item_field` from an inferred return annotation. That leaves explicit `response_model=...` generator routes internally inconsistent: they become JSONL streams, but the JSONL branch has no stream item field and claim 7 says to bypass validation with `jsonable_encoder`. This silently ignores the explicit `response_model`, unlike existing FastAPI response-model behavior. The spec should either derive the stream item field from an explicit response model for generator routes or reject explicit `response_model` on JSONL generator endpoints.

- `volley-round1.md:13` only requires JSONL routes to propagate dependency headers and background tasks, and `volley-round1.md:9` limits shared response-argument handling to normal and raw-streaming paths. This omits decorator `status_code` and dependency-mutated `Response.status_code` for default-response JSONL routes. Existing FastAPI response handling honors both, and the diff's JSONL path can easily regress to `200 OK` for `@app.get(..., status_code=201)` or for a dependency that sets `response.status_code`. The spec should require the same response-argument construction for JSONL streams, not just raw streams.

- `volley-round1.md:21` is too broad when it says any generator endpoint with an explicit `response_class` must pass the generator to that class as `content`. That is only valid for streaming response classes; explicit classes like `JSONResponse`, `PlainTextResponse`, or other non-streaming `Response` subclasses either cannot serialize a generator or would change existing FastAPI behavior that previously passed non-`Response` returns through `serialize_response`. The spec should scope the raw-generator pass-through to `StreamingResponse` subclasses or add a rejection for non-streaming explicit response classes.
