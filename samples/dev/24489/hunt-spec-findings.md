1. Claim 1 overstates the shared behavior between `mapParams` and `withUserHints`. The spec says both callers use the "single schema property or prompt" rule, but the current code does not: `AgentTool.mapParams` falls back to `{ prompt }` when the schema is not a record or has no `properties`, while `DelegateInvocation.withUserHints` returns the original args without appending hints unless `inputSchema.properties` is itself a record. See `packages/core/src/agents/agent-tool.ts:106` and `packages/core/src/agents/agent-tool.ts:213`. A helper that always returns `prompt` for non-one-property schemas would make remote agents with non-record schemas or schemas lacking `properties` start receiving user hints prepended to `prompt`, which is a behavior change. The spec needs to say whether the helper returns a key only when `properties` is present for the hint path, or whether this behavior change is intended.

2. Claim 4 does not specify the valid-JSON-but-not-object string case. The current `parseToolArguments` catches invalid JSON as an error, but for valid JSON strings that parse to an array, primitive, or `null`, it returns `{ args: {} }` with no error because only non-array objects are copied into `args`. See `packages/core/src/agents/local-executor.ts:1441`. The claim only names "string JSON object arguments" and "invalid JSON strings"; an implementer following "return parsed arguments directly" could accidentally return arrays/primitives as `args` or start erroring on them. The spec should explicitly preserve the existing `{ args: {} }` outcome for valid non-object JSON strings.
