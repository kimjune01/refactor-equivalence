# Phase 7 — Blind forced-choice review

You are reviewing a pull request. You will see two unlabeled diffs from the same base commit. Both versions pass the project's test suite.

## Context

- **PR title and description** are provided below.
- **Version A** and **Version B** are two implementations of the same change.
- Tests pass on both versions.

## Phase 1 — Forced choice

**Which version would you approve for merge?** You must pick one.

For each version, note:
- Does it raise any semantic concerns? (behavior that might be wrong despite passing tests)
- Brief rationale for your choice (1-2 sentences)

## Phase 2 — Trajectory classification

Now you will see a third version: **the one that was accepted after human code review.**

Compare your chosen version against this accepted version. Classify it:

- **Past:** simpler than the accepted version without loss of correctness or clarity
- **Short:** improved over the alternative but leaves complexity the accepted version removed
- **Wrong direction:** no meaningful improvement, or worse

## Phase 3 — Blinding check

Did you believe either version was:
- The final accepted version?
- Generated by an AI?
- Neither / couldn't tell?

## Output format

Write your response as JSON:

```json
{
  "forced_choice": "A" or "B",
  "rationale": "...",
  "semantic_concerns_A": "..." or null,
  "semantic_concerns_B": "..." or null,
  "trajectory": "past" or "short" or "wrong_direction",
  "trajectory_rationale": "...",
  "blinding": "identified_final" or "identified_ai" or "could_not_tell"
}
```
